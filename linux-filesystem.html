The Linux Filesystem

Under Construction!

Unix began life as a filesystem (Kernighan 33-34). Shortly thereafter, a means of interacting with it was created— initially, an exec function, a shell, text editor and assembler. Written entirely by Ken Thompson, these key components formed the nexus of the time-sharing operating system that would eventually be called Unix.

This file-first approach to operating system design is found in the implementation as well as the history of Unix and its derivatives; Unix and Linux use files for many things. As such, there is a lot to be said about files. This paper introduces the notion of a filesystem, and discusses the directory structure implemented with it.

Filesystems

A filesystem is a systematic method of organizing data on disk.[1][2] Conceptually, it is similar to a data structure, in that it defines a convention to be followed to store and retrieve data. Preparing a drive for use by a particular filesystem is known as formatting.

    Some Common Filesystems:

  • EXT4: Introduced in 2008,[1] EXT4 is the most common filesystem used on Linux today. It is an evolutionary enhancement on EXT3, supporting larger filesizes, deeper subdirectory support, faster handling of large files, along with many other improvements.[3] It is mature, fast, and stable.

  • ZFS: Introduced in 2005,[2] ZFS incorporates fancier features that are generally out of place on an ordinary Linux Desktop. It was developed by Sun Microsystems, and became closed-source with Sun's acquisition by Oracle.[2] It is intended for enterprise use, and is relatively easy to configure improperly. Also, it is not free.

  • FAT32: The "File Allcoation Table 32-bit" is a DOS-era filesystem,[3] and is widely supported, though antiquated. In particular, it is not a journaling filesystem, so that changes to data are not recorded before being carried out (increasing the likelihood of corruption in the event of, say, power outage). This format is often used to hold the bootloader, since it is virtually guaranteed to be supported by firmware.

Directory Structure

On top of a given filesystem, a directory structure is imposed by the operating system. On Linux and other Unix-like systems, the run-time directory tree begins at "root," which is denoted with a single forward slash, /. Ideally, this would form a mathematical tree, but in practice, this is rarely possible, so we must call it a directory structure. For Unix-like operating systems and its derivatives, directory structure is defined by the Linux Filesystem Heirarchy Standard; the standard is maintained by The Linux Foundation.

Within any GNU/Linux distribution, some common themes are found. For instance, bin directories are found in several places, such as /bin and /sbin, along with /usr/bin and often /home/user/bin. This system reports no less that 89 directories named "bin." Each of these contain executable files (programs), which can be ran as commands from the shell.

As per the standard, /etc (usually read "et-c") contains host-specific system configuration files; as a consequence, you can expect to mess around in this directory when configuring the system for the first time (or otherwise). It contains, for instance, files describing the configuration of the system's package manager, printers, and, well, etcetera.

Unlike Windows systems, which mount a new, distinct directory tree for each storage device on the system (C:\, D:\, etc.), Linux and other Unix-like systems mount new devices on the existing directory tree, using the mount command. Files within the freshly mounted filesystem are available at the location stipulated. The directory hierarchy has two distinct places to facilitate this, /mnt and /media; however, as super-user, one is free to mount a new filesystem just about anywhere. So, for instance,

$ sudo mount /dev/sda2 /home/josh

would mount the second partition of the device "sda" to /home/josh; its contents would then be available there.

In Unix, the directory tree is used as a general-purpose meeting place; it holds more than just regular files. The entities in /div, for instance, are called special device files (Linux Filesystem Heirarchy Standard, page 7); each file in /dev represents a device connected to the system (or an aspect of one). /dev/sda, as used above, refers to a SATA hard drive. /dev/sda1, refers to the first partition on that hard drive, /dev/sda2, refers to the second partition on that hard drive, etc..

This mechanism is general, extensible and allows both users and programmers to access the devices attached to the system easily (that is, it is obvious). For instance, what can you make from the results of,

$ ls /dev

?

While there is too much going on in the /dev directory to exhaust the subject here, it's worth noting that utilities for displaying the current system's hardware often begin with ls:

  • lsblk - List block devices (hardrives, CD-drives, etc)
  • lscpu - Display information about the CPU architecture
  • lsusb - List usb devices
  • lspci - List all PCI devices
  • lsof  - List open files
  • lsmod - Show the status of modules in the Linux Kernel

Pretty spiffy!

Regular Files

Regular files contain data. The contents of a regular file are not, in the general case, the business of the operating system. The operating system is tasked only with preserving data in a bit-perfect manner. That means that the Linux kernel treats text files, binaries, movies, source code, pdf's, etc., identically, and that any "special treatment," is implemented by user-space processes.

Approximately 80% of files are regular files (Stevens, p 98). We can verify this ourselves, by dividing the number of regular files,

$ sudo find / -type f | wc -l

by the number of total files,

$ sudo find / | wc -l

I arrive at 76%.

Each file on the system has nine mode bits: read, write and execute for each, owner, group, everyone else (GNU Coreutils, section 27). These are maintained by the filesystem along with other metadata about the file, such as size and location on-disk. For instance, the operating system will not even consider trying to execute a file if its appropriate executable bit is not set (but will allow you to pass the file as an argument to an executable).

Suppose a plaintext file has its executable bit set, and one tries to execute the file. It's clearly a script, right? The first line of such a script is often, #! /usr/bin/sh

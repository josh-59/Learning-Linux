The Linux Filesystem

Under Construction!

Unix began life as a filesystem (Kernighan 33-34). Shortly thereafter, a means of interacting with it was created— initially, this consisted of an exec function, a shell, text editor and assembler. Written entirely by Ken Thompson, these key components formed the nexus of the time-sharing operating system that would eventually be called Unix.

This file-first approach to operating system design is found in the implementation as well as the history of Unix and its derivatives: Unix and its derivatives use files for many things. As such, there is a lot to be said about files. This paper introduces the notion of a filesystem, and discusses the directory structure implemented with it.

Filesystems

A filesystem is a systematic method of organizing data on disk.[1][2] Conceptually, it is similar to a data structure, in that it defines a convention to be followed to store and retrieve data. Preparing a drive for use by a particular filesystem is known as formatting.

    Some Common Filesystems:

  • EXT4: Introduced in 2008,[1] EXT4 is the most common filesystem used on Linux today. It is an evolutionary enhancement on EXT3, supporting larger filesizes, deeper subdirectory support, faster handling of large files, along with many other improvements.[3] It is mature, fast, and stable.

  • ZFS: "Don't use ZFS. It's that simple." —Linus Torvalds.

    Introduced in 2005,[2] ZFS incorporates fancier features that are generally out of place on an ordinary Linux desktop. It was developed by Sun Microsystems, and became closed-source with Sun's acquisition by Oracle.[2] It was intended for enterprise use, and is relatively easy to configure improperly. Also, it is not free.

  • FAT32: The "File Allcoation Table 32-bit" is a DOS-era filesystem,[3] and is widely supported, though antiquated. In particular, it is not a journaling filesystem, so that changes to data are not recorded before being carried out (increasing the likelihood of corruption in the event of, say, power outage). This format is often used to hold the bootloader, since it is virtually guaranteed to be supported by firmware.

By and large, users of Linux don't often interact with the filesystem of their operating systems. The term was developed here to make a precise distinction between filesystem and directory structure.

Directory Structure

On top of a given filesystem, a directory structure is imposed by the operating system. On Linux and other Unix-like systems, the directory tree begins at "root," which is denoted with a single forward slash:

/

Ideally, the directory structure would form a pure mathematical tree, but in practice, this is rarely possible, so we must call it a directory structure. For Linux operating systems, this directory structure is defined by the Linux Filesystem Heirarchy Standard; the standard is maintained by The Linux Foundation.

So, within any GNU/Linux distribution, some common themes are found. For instance, bin directories are found in several places, such as /bin and /sbin, along with /usr/bin and often /home/user/bin. This system reports no less that 89 directories named "bin." Each of these contain binary executable files (programs), which can be ran as commands from the shell.

As per the standard, /etc (usually read "et-c") contains host-specific system configuration files; as a consequence, you can expect to fiddle with the contents of this directory when configuring the system for the first time (or otherwise). It contains, for instance, files describing the configuration of the system's package manager, printers, and, well, etcetera.

Unlike Windows systems, which mount a new, distinct directory tree for each storage device on the system (C:\, D:\, etc.), Linux systems place new filesystems within the existing directory structure, in an operation called mounting. Because the directory structure is not rooted on a drive, it is more abstract than those of Windows lineage; in addition, our directory structure is more stable over the duration of the machine's uptime.

The command to mount is mount, and the unmount command is umount. Both require superuser privledges.

Suppose you have a usb stick with some music on it, and you'd like to access it. After it's plugged into your computer, the hardware, then the kernel, recognize it, but it is unaccessible:

In practice, most distributions automatically mount usb sticks when they are plugged in. Moreover, no usb stick will be named /dev/usb; that was only a conceptual item. It's more likely to be named /dev/sdb1

In Unix, the directory tree is used as a general-purpose meeting place: It holds references to hardware drivers, information about running processes, your terminal screen is represented by a file within the hierarchy; all of this in addition to regular files. This mechanism is general, extensible and allows both users and programmers access to the running machine. The only trick is keeping it organized.

The entities in /dev, for instance, are called special device files (Linux Filesystem Heirarchy Standard, page 7); each file in /dev represents a device connected to the system (or an aspect of one). /dev/sdb, as used above, refers to a hard-disk driver. /dev/sdb1 refers to the first partition on that hard drive; /dev/sdb2 refers to the second partition on that hard drive; etc..

While there is too much going on in the /dev directory to exhaust the subject here, it's worth noting that utilities for displaying the current system's hardware often begin with ls:

  • lsblk - List block devices (hardrives, CD-drives, etc). Useful to determine what drive to mount.
  • lscpu - Display information about the CPU architecture
  • lsusb - List usb devices
  • lspci - List all PCI devices
  • lsof  - List open files
  • lsmod - Show the status of modules in the Linux Kernel

Regular Files

Regular files contain data. The contents of a regular file are not, in the general case, the business of the operating system. The operating system is tasked only with preserving data in a bit-perfect manner. That means that the Linux kernel treats text files, binaries, movies, source code, pdf's, etc., identically, and that any "special treatment," is implemented by user-space processes.

Approximately 80% of files are regular files (Stevens, p 98). We can verify this ourselves, by dividing the number of regular files,

$ sudo find / -type f | wc -l

by the number of total files,

$ sudo find / | wc -l

I arrive at 76%.

Each file on the system has nine mode bits: read, write and execute for each, owner, group, everyone else (GNU Coreutils, section 27). These are maintained by the filesystem along with other metadata about the file, such as size and location on-disk. For instance, the operating system will not even consider trying to execute a file if its appropriate executable bit is not set (but will allow you to pass the file as an argument to an executable).

Aside: Shebang

Suppose a plaintext file has its executable bit set, and one tries to execute the file. It's clearly a script, right? The first line of such a script is often, #! /usr/bin/sh

The mode bits can be changed with the chmod utility. A common operation is setting the executable bit: chmod +x: Change the way the operating system handles a file; in addition to the above, the operating system can now "execute" the file: Binary #! /usr/bin/XXXXXXX Bash's behavior by default For instance, the ruby interpreter is a program, existant in my filesystem as a binary file compiled from C source code; to execute a ruby program, I pass the argument ruby my_example_program.rb to bash, causing the kernel to launch ruby with the argument file my_example_program.rb.

Processes

"Theory and practice sometimes clash.
And when that happens, theory loses. Every single time.
—Linus Torvalds

Aside:

Recall that a file is a sequence of bytes.

A process is an instance of an executing program.[1][2] A program (also known as an "executable file") contains the information required to construct a process.[1] The process is, after the file, the most important abstraction of operating systems.

Actors

In Unix and its derivatives, processes are the actors of the system: Left to its own devices, the kernel does nothing. This is a defining feature of the Unix architecture— All Unix-like operating systems depend upon processes to direct the kernel, and through the kernel, processes direct the machine. The result is a clear demarcation between what to do (dictated by processes), and how to do it (dictated by the kernel).

The kernel is responsible for:

  1. Facilitating new process creation
  2. Allocating hardware resources to each process
  3. Maintaining information about processes
  4. Facilitating change of their attributes

There are processes for every function on the system. There's one for the network manager, another for the desktop environment, another acting as a printer spooler, Firefox runs as a process, etc..

Under the Unix philosophy, each process has a narrowly defined role, which it is expected to do well. It might copy a file, list the contents of a directory, provide audio services, or browse the internet. It is intended that these programs work together to accomplish any given task, thereby increasing the modularity of the whole, and, consequently, easing programming and maintenance.

Runlevels

Historically, collections of running processes were called runlevels, and were managed by the initialization ("init") process. Different runlevels offered different services, and therefore were intended to be used in different situations. For instance, single-user mode (runlevel 1) was analogous to safe boot, runlevel 5 corresponded to multi-user/GUI mode, and runlevel 0 told init to kill all processes and shut down the system.[3]

Process Creation

and System Startup

At the end of the boot sequence, the kernel launches a single program, /sbin/init,[4] and in so doing creates the system's first process. After /sbin/init begins execution, process creation is delegated to processes themselves. While init schemes have evolved over the years, its purpose has not changed: It is a process with root-level permissions that acts as a manager of the system and trustee among processes.

Aside:

You can check your init program with

$ ps --pid 1

systemd is very common today.

init's first responsibility is to bring up userspace. It does this by launching a sequence of programs, most of which act as services, some of which the user interacts with directly. After this, it operates in the background, managing these services, restarting them if they falter, and logging events for administrative purposes. Its final responsibility is to shut the system down cleanly.

From within a process, launching another program is broken into two steps: Duplicate the current process, then instruct the new process to execute a different program.

The first step is to copy the current process using the fork(2) system call. This system call instructs the kernel to duplicate the current process; the two resulting processes are nearly identical.[4] Of note, they both continue execution of the same program at the same spot, they have identical (but separate) memory spaces, and they have the same files open.

The newly created process is referred to as the child process, the old one is referred to as the parent process, and we say that the child process inherits its parent's attributes. Of the ways that parent and child differ, one is important to us now: They each recieve a different return value from fork(). This allows them to act differently within the same program:

Note:

Any process may call fork(). 

        pid_t rv = fork();
        if(rv == 0) {
            // I am child

        } else if (rv > 0) {
            // I am parent

        }

The second step is to instruct the child to launch a different program using execve(2) or one of its variants. This instructs the kernel to replace the currently executing program with another; therein, execution begins at main().

Aside:

Any process may call execve(). 

        pid_t rv = fork();
        if( rv == 0 ) {
            // I am child
            char *args[] = {"ls", "-l", NULL};
            execve("/usr/bin/ls", args, NULL);

        } else if ( rv > 0 ) {
            // I am parent
            return 0;

        }

This sequence is traditionally called fork and exec, and results in one process (the parent) launching another program (the child). It was present in Ken Thompson's original implementation of Unix at Bell Laboratories, and for reasons we'll see later, is ideal for programs at the command-line.

Finally, note that each process, less init, has a unique parent process. As such, the collection of processes on the system forms a tree, with init at root.

Process Attributes

Upon birth, each process is given its own process id (PID) number by the kernel. This is guaranteed to be unique to the process, and remains constant throughout the process's life; it serves as the system's handle to the process.

For each process on the system, the kernel maintains a great deal of information, including:

Aside:

For detailed information about processes in the Linux environment, see The Linux Programming Interface.

  • Location of executable file
    • Location of next instruction within
  • Files currently open by the process
    • Access Mode (read, write, read and write)
    • Location wihin file
  • Current working directory
  • Owner, group
  • Environment variables

Most of this information is available for perusal in the /proc/ directory, which contains one subdirectory for each process on the system. In order to give some idea of how the kernel manages processes, and how processes interact with the kernel, we will consider one of these attributes: Files opened by processes.

File Descriptors

Each file that a process has open is assigned a number by the kernel (specifically, a small integer). This number is called a file descriptor. These integers are used by the process whenever the process requests an operation on a file. In the following call to read(2), the calling program is instructing the kernel to read from the file associated with the number 3:

num_bytes_read = read(3, buff, buff_size);

Note:

You are not expected to memorize these interfaces. The manual page for read(2) describes each field, and the kernel's response, in detail.

The kernel will read at-most buff_size bytes from the file associated with file descriptor 3, and store the data into buff. If all goes well, the system call will return with a nonnegative value; if the kernel encountered an error, it will return with a negative value. In either case, this return value will be assigned to num_bytes_read.

Write(2) is handled analogously. Every system call has a well-defined and stable interface, and an entry in the man pages.

Process Termination

A process may terminate for a few reasons:

  • Return from main()

    This is the usual and preferred route to terminate a process: It finishes its job, and explicitly requests to return.

    When this occurs on the command line, the process's return value (and it must offer one) is available for inspection. By convention, a return value of zero indicates success, while nonzero return values indicate error or fault.

  • It is told to terminate

    A process may be told to terminate by any other process with the same owner. A process owned by root (that is, a process with superuser priveledges) can instruct any process to terminate.

    Command-line tools for terminating processes are kill(1), which refers to processes by their PID numbers, and killall(1), which kills processes by name.

  • It does something wrong

    Almost always, this happens when the offending process tries to read or write a memory location not assigned to it, a condition known in Linux as segmentation fault.

    Note that when this occurs, it is not the case that the process is trying to read or write some other process's memory. Rather, every process is given its own virtual memory address space (see below); the kernel kills it because the memory in question is not associated with the process in any meaningful way.

Sandbox

Processes are one of the big ideas of operating systems. Through it, operating systems can isolate every running program from every other running program, and from the machine itself. Effectively, every program on the machine executes in a sandboxed environment, composed of restricted access to the CPU and a distinct memory space.

Limited Direct Execution

Modern CPU architectures support two distinct modes of operation. Kernel mode, sometimes called "supervisor mode," is the unrestricted mode. Here, all CPU instructions are available, so that code running in this mode has complete access to the machine.

User mode is a proper subset of kernel mode; code running in user mode does not have access to all instruction. All processes run in user mode. If a process, running in user mode, attempts to execute a restricted instruction, its execution is halted, and control is given to the kernel. At that point, it is up to the operating system to determine what to do (usually, terminate the offending process).

This technique is known as limited direct execution.[3] It is limited in the sense that each process has access to only a restricted subset of the CPU's instructions. It is direct in the sense that each process runs directly on hardware. It makes safe and efficient use of available hardware.

For instance, no process can access a hard-drive directly. The proper avenue for a process to access such a device is through the kernel, via system calls. Through limited direct execution, this is the only way that a process can access it. This ensures that the kernel remains in charge of the machine's state.

Virtual Memory

Modern operating systems provide each process with its own memory in the form of a distinct address space. An address space is a collection of memory addresses; on all modern architectures, memory (RAM) is byte-addressable, so that each address refers to exactly one byte. Addresses themselves are simply numbers, usually expressed in hexadecimal.

Some archaic operating systems (such as the Amiga) allowed processes direct access to the system's main memory; processes effectively shared the same address space. This posed significant issues for system stability and security— namely, one process could read and/or manipulate the memory of another.

Modern operating systems provide each process with a distinct address space. The result has been a profound improvement in system stability. The addresses that any process uses are relative to that process alone; they are called "virtual" because they exist, effectively, for that process alone. These virtual addresses are mapped onto physical addresses by the CPU, without the process ever being aware of it. This scheme is referred to colloquially as "virtual memory."

The virtual memory abstraction requires support from both the kernel and CPU. For each process, the kernel maintains an address translation table, which maps virtual addresses onto physical ones. This is loaded onto the CPU's Memory Management Unit whenever the process runs, and the MMU carries out address translation automatically whenever in user mode.

Crucially, not every virtual address needs to be mapped to a physical address. Rather, when a process references a virtual address that is not mapped, the MMU instructs the CPU to interrupt execution of the process; the CPU defers control to the kernel. If the memory address should be mapped, the kernel loads that data into memory, modifies the translation table and returns control to the process. If the memory should not be mapped (i.e., it is an undefined address), the kernel aborts execution of the program.

Through the mechanism of virtual memory, each process is presented with the illusion that it has the entirety of the machine's memory to itself. Other processes are implicitly protected from memory tampering: Their memory simply does not exist within the context of the currently running process, and therefore cannot be read from, nor written to. As with limited direct execution, virtual memory facilitates the safe and efficient use of a machine resource.

Learning Linux

A Tour Through Documentation

Linux is a very large subject. Although there are many sources of documentation, they are not, in general, well-organized, so that finding the right information can be hard. This paper aims to give an overview of available resources, some examples of each, and useful pointers, where appropriate.

How-to

It's one of the great failings of modern computer culture to suggest that being new or trying to learn is embarassing.

  1. Figure out what your question is.

    This can be more difficult than you might guess, because you might honestly not know how to phrase your question. Be specific! Things are very well-defined in computers. This stands in great contrast to intuitive conceptions of their operation.

  2. What would answer your question?
    • An Example
    • A Precise Explanation
    • A Detailed Description
    • Theoretical Discussion

  3. Revise.

    Do you need more information? Or less? You might need something immediately useful rather than theoretically correct. Conversely, you might find that your practical knowledge is lacking some kind of body; in this case, you are looking to frame your knowledge in appropriate theory.

  4. Be Patient.

    Skill accrues. Answer one question at a time.

Man Pages

By far the most useful asset in the Linux documentation chain is the manual ("man") pages. They are concise, immediate, and generally aim to answer as many questions as quickly as possible. However, they tend to be more "handy" than "exhaustive," so they might not answer your question.

Man pages are accessed at the command line so:

man app

You can search through available man pages with the apropos command, as in,

apropos decompress

Technically speaking, this utility searches the single-line, "short description" of each man page, and prints out the names and descriptions of programs that match your search (man apropos for more information).

Once within the man utility, you can search the current page by typing a forward slash, then your search term:

/<search-term>

Then type a single n to jump to the next result. Many of the controls of the man-pages coincide with those of the less paging utility, and with those of the vi/vim text editor, so they're worth learning. For instance, move down the page one line by typing j. Move up the page one line by typing k. And, move down a screenful by hitting the spacebar.

Manual Sections:

  1. Shell Commands
  2. Systems Calls
  3. C Library Functions
  4. Special Files
  5. File Formats and Conventions
  6. Games
  7. Miscellaneous

Man pages date back to the first implementations of Unix,[1] where they were the user's primary documentation of the newly-minted Unix operating system. They often include examples, hints, and useful suggestions (such as "These are not the interfaces you are intered in.").

The structure of the manual is worth learning. For instance, the manual is divided into sections, so that just by looking at the location of a page within the manual, one can tell if it's a program that can be called from the command-line, a system call, or a C library function; man man for more information.

Similarly, knowing the structure of the man-pages themselves has proven useful. For instance, each page has a "See Also" section near the bottom, which lists similar or related man pages.

Official Documentation

Official documentation originates from the group or organization that is producing the work. It is suitably positioned to be both exhaustive and authoritative over the subject at hand. Think of offical documentation like the primary sources from history class. This is by far my favorite form of documentation. Nothing makes me feel warmer and fuzzier than downloading the Official ________ _________ Reference Manual.

However, it's unreasonable to expect people to produce such technically meritorious writing in multiple formats simultaneously, such as HTML, PDF, etc. (although GNU does an admirable job). As a consequence, official documentation is typically found in web-page format. Moreover, like the projects they are part of, official documentation is often community-driven.

Several awesome examples spring to mind:

  • The Rust Progamming Language— Written by Steve Klabnik and Carol Nichols with updates from the community, this book is the main documentation of the Rust language. I think it's a great book, especially for its technical excellence.
  • Godot Open Source Game Engine— Godot is an MIT-licensed open source game engine (and, I daresay, the one you're looking for). Its official documentation is community-driven.
  • GNU's official documentation is both supremely authoritative and exhaustive.
  • Arch Linux's Wiki is great, and almost always applies to distributions outside of Arch.
  • Mozilla's HTML Docs— Mozilla produces excellent documentation, primarily about web technologies. In addition, they help define the HTML living standard,[2] and hence, represent an official voice with respect to modern HTML.

Unofficial Documentation

Most documentation is from unofficial, secondary sources. Examples include this website, websites like it, question/answer sites, books that are about Linux without being authoritative (which is most of them), Youtube videos, podcasts, etc.. Unofficial documentation is very hit-or-miss, and whether it represents a hit for you is largely a matter of taste. Think of them like commentary on the subject at hand.

Examples of good, unofficial documentation:

  • The Linux Documentation Project— Sadly, this project is not maintained anymore.
  • The Linux Command Line by William Shotts— An Introduction to Linux.
  • Understanding Unix/Linux Programming: A Guide to Theory and Practice by Bruce Molay— A personal favorite, though it is dated. It walks the reader through the structure of Linux via programming tutorials.

There are two big problems with this sort of documentation. First, it isn't necessarily authoritative. Since it doesn't spring from the source, it is not moderated by the creators of the project. Second, there's there's just too much of it. As my professor said, "There are a gazzillion books on Linux."

Tutorials

Tutorials are usually unofficial. That being said, they are a great form of unofficial documentation, because they are informative, concise, and relevant. A tutorial jumps right in to the subject at hand. However, it's possible to produce a tutorial that is impractical, such as, How to Write a Flight Simulator From Scratch.

Good examples:

  • Make an Action RPG in Godot 3.2— A Youtube tutorial for building your own Godot 3.2 game (note, the Godot supplied via Debian repository was outdated, at the time of this writing [April 2020], so you might need to check your version to get it to work.).
  • Learn Markdown— A tutorial guide to markdown.
  • Making Your Own Ruby Gems — Walks you through the creation of a Ruby software package (a.k.a., 'gem').

Videos

Again, these are usually unofficial. Videos are addictive because they're easy to digest, but real understanding only comes out of real study— Read, study, do!

Some channels I like:

  • Engineer Man— Linux-related programming content. Engineer Man got me off the couch and into some Linux; he's awesome!
  • Level1Techs— Tech news, mostly.
  • DistroTube— Linux distro's, applications.

Reference

There exists one last category of documentation, reference material. Technically speaking, it's unofficial documentation. That can't stop someone from doing their homework (so to speak). Reference material is characterized by being well-researched. Good reference material is characterized by being laid out well, and presenting material that is useful to you.

Some Suggestions:

  • The Linux Programming Interface by Michael Kerrisk— My recommendation
  • The Design of the Unix Operating System by Maurice J. Bach— Old-school, somewhat dated.
  • Advanced Programming in the Unix Environment, by W. Richard Stevens and Stephen A. Rago— Highly theoretical, it covers the programming interfaces (system calls) of various standards (POSIX, Single Unix Specification, etc..).

These are enormous books, most appropriate to the professional, the aspiring professional, or the insanely curious. As my professor said, these are the sorts of books that you read over the course of six months or a year, but which can define your career.

Programming Theory

I love books, especially free ones. Some favorites:

  • Fundamental Concepts in Programming Languages by Christopher Stauchey.
  • Open Data Structures by Pat Morin— Best I've ever read on data structures.
  • Discrete Mathematics by Oscar Levin— Very clear and lucid presentation.

Though foreign to most users, the text-terminal is the original interface for "interactive computing." This article will examine the fascinating theory of terminals and shells in GNU/Linux. For a tutorial introduction, see the Ubuntu Linux Command Line for Beginners.

Physical Terminals

A Brief History

A physical terminal was a computer peripheral similar to a typewriter. The earliest examples printed output onto paper; this technology was carried over from the teletypewriter, a device for sending telegrams. Paper-based terminals persisted well into the 1970's, and later electronic typewriters sometimes supported terminal mode, allowing them to interface with computers via serial cable.

Ken Thompson (sitting) and Dennis Ritchie at PDP-11, in Bell Labs' UNIX Room[1]

As technology developed and interactive computing became more popular, hard-copy terminals were phased out in favor of screen-based entities. The resulting devices consisted of a monitor and keyboard.

The monitor contained a buffer (on the order of a few kilobytes) which maintained the text currently on-screen.[2] Using an onboard font table, it would redraw the text given to it without further instruction by the computer; thus, the computer's only duty was to feed the terminal characters.

  • Hyper— An Electron-based terminal

These emulators are "dumb" in the sense that they do nothing except display text and return text. Per tradition, terminals are represented as files.

Terminals can become quite involved; for this introduction, and for most purposes, it is sufficient to recognize that terminals and shells are distinct, cooperating processes.

Shells

Aside:

Desktop environments serve a role analogous to shells.

A shell is a program which interprets and executes commands. It is a shell in the sense that it is the outermost layer of the operating system, and the one which the user has direct contact.[a]

The UNIX architecture entertains the shell as an ordinary user-space process: From its perspective, it is just another running program.

Some modern shells:

  • Bash— A GNU Production, pretty much the standard shell
  • Zsh— A shell "designed for interactive use; very similar to Bash.
  • Xonsh— A Pythonic shell, written in Python
  • Fish— The Friendly Interactive Shell
  • Nushell— An interactive shell, supporting piplines of structured data, in addition to simple text streams.
  • Elvish— an "expressive programming language and versatile interactive shell." Version 1.0 not yet released

Commands

A typical line of input to a shell has the following form:

$ command [arg ...]

This corresponds directly to execute another program named command, and pass it the arguments in [arg ...].

The shell, for its part, finds the program, forks itself and executes that command. This is canonical shell behavior, and all programs that can be implemented in this manner are.

For example, one might expect the well-used command ls to be built in to the shell, and in lesser constructs, that would be the case! In fact, ls exists as /usr/bin/ls, and the shell dutifully locates and executes it each time we ask to see a directory's contents. Similar remarks apply to any command that does something, be it move a file, delete a directory, or view the system's processes.

Aside:

If you're not interested in our command-line, don't worry: Distributions exist that don't require its use.

The result is a highly programmer-centric paradigm. To create a new command just became very simple: Put your program in a location that the shell will look, and then you may execute it natively, as though it had been there all along— No need to rebuild the shell.

In the Beginning

The first working versions on UNIX entertained exactly two processes: One for each of two terminals connected to the machine.[3] The shell, sh, existed as a user-space process, but fork() had not been introduced. To execute a different program, the shell simply replaced itself with the requested program; the program, on calling exit, replaced itself with the shell once again.[3]

PATH Environment Variable

The PATH environment variable contains a list of directories to be searched when looking for a command. This list is colon (:)–separated, and may look like,

$ echo $PATH
/usr/bin:/bin:/home/josh/.local/bin

The shell searches each directory in this list from left to right for a file whose name matches the command given. It executes the first found, and then abandons its search.

Some caveats apply: If the command is a builtin, Bash will execute the builtin, and ignore PATH. More, Bash does not search the current working directory for a command.

We can overrule both builtins and the shell's search of PATH including a forward-slash / in our command, typically:

$ ./command

Appending locations to PATH is an important step of setting up a development environment. This is typically done in ~/.bashrc. The correct construct appends the directory only if it is not already present:

if [[ ! $PATH =~ $HOME/.local/bin ]]; then
    PATH=$PATH:$HOME/.local/bin
fi

Working Directory

Recall that each process has a current working directory. Since the shell is a process, naturally, it too has a current working directory. At the command line, this is of particular importance: It reflects where you are in the directory tree.

The shell's current working directory may be printed with the command pwd:

$ pwd
/home/josh

Its value may be changed with cd:

$ cd Public
$ pwd
/home/josh/Public

cd is a builtin

In Ancient UNIX, change directories was implemented as a separate command. When invoked, it inherited the shell's current working directory, changed it, and then the shell would inherit that working directory. With the inclusion of the fork() system call, though, the chdir command broke:

"There was much reading of code and anxious introspection about how the addition of fork could have broken the chdir call. Finally the truth dawned: in the old system chdir was an ordinary command; it adjusted the current directory of the (unique) process attached to the terminal. Under the new system, the chdir command correctly changed the current directory of the process created to execute it, but this process promptly terminated and had no effect whatsoever on its parent shell! It was necessary to make chdir a special command, executed internally within the shell. It turns out that several command-like functions have the same property, for example login."
—Dennis M. Ritchie
The Evolution of the Unix Timesharing System

The cd command is also, necessarily, built in to the shell. Commands such as these are today called builtins, and can be investigated by:

$ help

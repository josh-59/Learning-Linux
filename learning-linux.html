Learning Linux

"Understanding is the key to success with Linux."
Linux System Administrator's Guide

Linux is a very large subject. Although there are many sources of documentation, they are not, in general, well-organized, so that finding the right information can be hard. This paper aims to give an overview of available resources, some examples of each, and useful pointers, where appropriate.

How-to

  1. Figure out what your question is.

    This can be more difficult than you might guess, because you might honestly not know how to phrase your question. Be specific! Things are very well-defined in computers. This makes computers unique.

  2. What would answer your question?
    • An Example
    • A Precise Explanation
    • A Detailed Description
    • Theoretical Discussion

  3. Revise.

    Do you need more information? Or less? You might need something immediately useful rather than theoretically correct. Conversely, you might find that your practical knowledge is lacking some kind of body; in this case, you are looking to frame your knowledge in appropriate theory.

  4. Be Patient.

    Skill accrues. Answer one question at a time.

Man Pages

By far the most useful asset in the Linux documentation pool is the manual ("man") pages. Man pages date back to the first implementations of Unix,[1] where they were the user's primary documentation of the newly-minted operating system. They often included examples, hints and useful suggestions, such as, "cat file is about the easiest way to print a file."

Today, the man pages are maintained by Michael Kerrisk, and often include examples, hints, and useful suggestions, such as, "These are not the interfaces you are interested in." Learn to use man pages early, for they are a way of life.

Manual Sections:

  1. Shell Commands
  2. Systems Calls
  3. C Library Functions
  4. Special Files
  5. File Formats and Conventions
  6. Games
  7. Miscellaneous

Man pages are accessed at the command line so:

man app

You can search through available man pages with the apropos command, as in,

apropos decompress

or

apropos extract

Technically speaking, this utility searches the single-line, "short description" of each man page, and prints out the names and descriptions of programs that match your search (man apropos for more information).

Once within the man utility, you can easily search the content by typing a forward slash, then your search term:

/<search-term>

Then, enter. To jump to the next result, type a single n.

Many of the controls of the man-pages coincide with those of the less paging utility, and the vim text editor, so they're worth learning. For instance, to move down one line, type j. To move up one line, type k. And, move down a screenful by hitting the spacebar. Why the crazy controls? It keeps your fingers on the home row.

The manual is structured into sections, as given above. In particular, Section 1 contains entries for user commands, such as cat, man, mv and cp, while Section 3 contains the manual pages for C library functions, such as getline(), printf() and strtol(). When using apropos, one can restrict the search to a particular section with the -s switch:

apropos -s 3 printf

Otherwise, apropos lists them all.

In addition, each manual page is organized into a common format. For instance, each page has a "See Also" section near the bottom, which lists related commands/library functions.

Official Documentation

Official documentation originates from the group or organization that is producing the work. It is suitably positioned to be both exhaustive and authoritative over the subject at hand. Think of offical documentation like the primary sources from history class. This is by far my favorite form of documentation. Nothing makes me feel warmer and fuzzier than downloading the Official ________ _________ Reference Manual.

However, it's unreasonable to expect people to produce such technically meritorious writing in multiple formats simultaneously, such as HTML, PDF, etc. (although GNU does an admirable job). As a consequence, official documentation is typically found in web-page format. Moreover, like the projects they are part of, official documentation is often community-driven.

Several awesome examples spring to mind:

  • The Rust Progamming Language— Written by Steve Klabnik and Carol Nichols with updates from the community, this book is the main documentation of the Rust language. I think it's a great book, especially for its technical excellence.
  • Godot Open Source Game Engine— Godot is an MIT-licensed open source game engine. Its official documentation is community-driven.
  • GNU's official documentation is both supremely authoritative and exhaustive.
  • Arch Linux's Wiki is great, and almost always applies to distributions outside of Arch.
  • Mozilla's HTML Docs— Mozilla produces excellent documentation, primarily about web technologies. In addition, they help define the HTML living standard,[2] and hence, represent an official voice with respect to modern HTML.

Unofficial Documentation

Most online documentation is from unofficial, secondary sources. Examples include this website, websites like it, question/answer sites, Youtube videos, podcasts, etc.. Think of unofficial documentation as commentary on the subject at hand.

Examples of good, unofficial documentation:

  • The Linux Information Project — LINFO emphasizes useful, accurate, clear, consistent and interesting documentation of Linux[3]. It's a neat site, but was last updated in 2007.
  • The Linux Documentation Project— Sadly, this project is not maintained anymore, so that its content is often out of date.
  • The Linux Programming Interface by Michael Kerrisk— The de-facto reference for programming in Linux.

A piece of advice: Stay away from books. The problem is that there's just too many of them, and better sources are often available. As my professor said, "There are a gazzillion books on Linux." (and I own, like, ten). They tend not to be authoritative; they tend to be *somewhat* out-of-date. Moreover, there's just no need to pay for a secondary source. I, personally, would rather read the Official GNU Bash Reference Manual than any secondary source.

There's an exception to every rule. Michael Kerrisk is currently the maintainer of the Linux Man Page project; his book, The Linux Programming Interface, is an excellent text on the structures of the Linux kernel and the programming interface that it offers.

Tutorials

Tutorials are a great form of documentation, because they are informative, concise, and relevant. A tutorial jumps right in to the subject at hand.

Good examples of good tutorials:

  • Make an Action RPG in Godot 3.2— A Youtube tutorial for building your own Godot 3.2 game.
  • Learn Markdown— A tutorial guide to markdown.
  • Making Your Own Ruby Gems — Walks you through the creation of a Ruby software package (a.k.a., 'gem').

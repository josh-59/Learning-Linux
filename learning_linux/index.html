Learning Linux

A Tour Through Documentation

Linux is a very large subject. Although there are many sources of documentation, they are not, in general, well-organized, so that finding the right information can be hard. This paper aims to give an overview of available resources, some examples of each, and useful pointers, where appropriate.

How-to

It's one of the great failings of modern computer culture to suggest that being new or trying to learn is embarassing.

  1. Figure out what your question is.

    This can be more difficult than you might guess, because you might honestly not know how to phrase your question. Be specific! Things are very well-defined in computers. This stands in great contrast to intuitive conceptions of their operation.

  2. What would answer your question?
    • An Example
    • A Precise, Detailed Explanation
    • A Description
    • Theoretical Comprehension

  3. Revise.

    Do you need more information? Or less? You might be searching for something practically useful rather than theoretically exact. Conversely, you might find that your practical knowledge is lacking some kind of body; in this case, you are looking to frame your knowledge in appropriate theory.

Man Pages

By far the most useful asset in the Linux documentation chain is the manual ("man") pages. They are concise, immediate, and generally aim to answer as many questions as quickly as possible. They often include examples, hints, and useful suggestions (such as "These are not the interfaces you are intered in.").

They are accessed at the command line so:

man app

You can search through available man pages with the apropos command, as in,

apropos decompress

Technically speaking, this utility searches the single-line, "short description" of each man page, and prints out the names and descriptions of programs that match your search (man apropos for more information).

Once within the man utility, you can search the current page by typing a forward slash, then your search term:

/<search-term>

Then type a single n to jump to the next result. Many of the controls of the man-pages coincide with those of the less paging utility, and with those of the vi/vim text editor, so they're worth learning. For instance, move down the page one line by typing j. Move up the page one line by typing k. And, move down a screenful by hitting the spacebar.

Manual Sections:

  1. Shell Commands
  2. Systems Calls
  3. C Library Functions
  4. Special Files
  5. File Formats and Conventions
  6. Games
  7. Miscellaneous

Man pages date back to the first implementations of Unix,[1] where they were the user's primary documentation of the newly-minted Unix operating system. They are filled with useful notes, but tend to be more "handy" than "exhaustive," and often refer the reader to external documentation.

The structure of the manual is worth learning. For instance, just by the location of a page within the manual, one can tell if it's a program that can be called from the command-line, a system call, or a C library function; man man for more information. Similarly, the structure of the man-pages themselves is worth learning. For instance, each page has has a "See Also" section near the bottom, which is useful for finding out what else is available.

Official Documentation

Official documentation originates from the group or organization that is producing the work. It is suitably positioned to be both exhaustive and authoritative over the subject at hand. Think of offical documentation like the primary sources from history class. This is by far my favorite form of documentation— Nothing makes me feel warmer and fuzzier than downloading the Official ________ _________ Reference Manual.

However, it's unreasonable to expect people to produce such technically meritorious documentation in multiple formats simultaneously, such as HTML, PDF, etc. (although GNU does an admirable job). As a consequence, official documentation is typically found in web-page format (and is often community-driven).

Several awesome examples spring to mind:

  • The Rust Progamming Language— Written by Steve Klabnik and Carol Nichols with updates from the community, this book is the main documentation of the Rust language. I think it's a great book, especially for its technical excellence.
  • Godot Open Source Game Engine— Godot is an MIT-licensed open source game engine (and, I daresay, the one you're looking for). Its official documentation is community-driven.
  • GNU's official documentation is both superemely authoritative and technically exhaustive.
  • Arch Linux's Wiki is great, and almost always applies to distributions outside of Arch.
  • Mozilla's HTML Docs— Mozilla helps to define the HTML living standard,[2] and hence, represents an official voice with respect to modern HTML.

Unofficial Documentation

Most documenation is from unofficial, secondary sources. Examples include this website, websites like it, books that are about without being authoritative (which is most of them), Youtube videos, podcasts etc.. Unofficial documentation is hit-or-miss, and whether it represents a hit for you is largely a matter of taste. Think of them like commentary of the subject at hand.

Examples of good, unofficial documentation:

  • The Linux Documentation Project— Sadly, this project is not maintained anymore.
  • The Linux Command Line by William Shotts— An Introduction to Linux.
  • Understanding Unix/Linux Programming: A Guide to Theory and Practice by Bruce Molay— Personal favorite. It walks the reader through the structure of Linux via programming tutorials.

Honestly, the biggest problem with this sort of documentation is that there's just too much of it. As my professor said, "There are a gazzillion books on Linux." Moreover, the above sources (and the many like them) are not necessarily authoritative.

Theory

I put "theory" into its own category in order to avoid discussions about "university" (which deal primarily in theory). Theory is worth studying insofar as it yields practical insight (or entertainment). I like studying languages and mathematics, myself.

To get your feet wet:

  • Fundamental Concepts in Programming Languages by Christopher Stauchey.
  • Operating Systems: Three Easy Pieces by Remzi H. Arpaci-Dusseau and Andrea C. Arpaci-Dusseau.
  • Open Data Structures by Pat Morin— If you're pursuing a career in software development, you must learn data structures (I know, right?).

Tutorials

This. Probably the best form of unofficial documentation is the tutorial, because it is informative, concise, and relevant. Moreover, it jumps right in to the subject at hand.

Good examples:

  • Make an Action RPG in Godot 3.2— A Youtube tutorial for building your own Godot 3.2 game (note, the Godot supplied via Debian repository was outdated, at the time of this writing [April 2020], so you might need to check your version to get it to work.).
  • Learn Markdown— A tutorial guide to markdown.
  • Making Your Own Ruby Gems — Walks you through the creation of a Ruby software package (a.k.a., 'gem').

Videos

Some channels I like:

  • Engineer Man— Linux-related programming content. Engineer Man got me off the couch and into some Linux; he's awesome!
  • Level1Linux— Very cool channel, one of my favorites.
  • DistroTube— Linux distro's, user-space applications.

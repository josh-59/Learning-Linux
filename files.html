Files

"Theory and practice sometimes clash.
And when that happens, theory loses. Every single time.
Linus Torvalds

The protogenesis of Unix was a filesystem.[1][2] Designed by the researchers of Bell Laboratories and implemented primarily by Ken Thompson, this piece of software became an operating system when a means of interacting with its contents was introduced. Being a Unix-like operating system, GNU/Linux inherits Unix's file semantics.

Files

In Unix, a file is a sequence of bytes.[1] This definition is sufficiently abstract to allow, for instance, a keyboard to appear as a file. Unix-like kernels facilitate interaction with files via four essential system calls:

  1. open()
  2. close()
  3. read()
  4. write()

In particular, the read() system call fills a given buffer with the contents of a file; order is preserved across sequential calls to read(). When a program requests a read, the kernel suspends its execution, loads the appropriate data, then resumes execution of the program.

File Attributes

Unix-like operating systems characteristically store a substantial amount of information about each file on the system. This information, known as metadata, is stored alongside the file, typically on-disk.

Aside:

Oddly, this information does not include the file's name. More on this below.

File metadata consists of:

  • File Type
  • Read/Write/Execute Permissions
  • Owner
  • Group
  • Size
  • Timestamps

This information is kept for every file on the machine, and can be explained in part by the operating system's multi-user heritage. It is used today to compartmentalize the machine.

Device Files

The Unix directory tree is used as a general-purpose meeting place: In addition to regular files, it also contains references to hardware, and allows interprocess communication. This mechanism is general, extensible, and allows both users and programmers access to the running machine.

A device file represents a device connected to the system. They reside in /dev.[3] Each file therein serves as a user-space handle to its respective device. For example, when a USB-stick is inserted, a new file appears in /dev, and it may be read from or written to directly by passing this file to tools like cat or dd.

Note that the essential capabilities of the USB-stick are preserved through the file interface: It can be written to and read from. If we were to add the capacity to seek (i.e., change location) within the file, then we would be able to use the device in its entirety, through this interface alone.

Block and Character Device Files

Unix kernels support two kinds of device files: Block and character. A character device file may only be written to, or read from, a single byte (ASCII character) at a time. A terminal is one kind of device represented by a character device file.

Hard disks, SSD's and removable mass media are represented by block device files. Since the underlying devices support reading and writing in blocks of bytes at a time, the kernel must take care to buffer arbitrary user-space interactions with it. Additionally, block device files support seeking, whereas character devices do not.

Using Block Special Device Files

Operations involving block device files are relatively rare, and require superuser permissions. Given a new or blank block device, the typical course is to first partition the drive with fdisk or parted, then to create a filesystem on the partition (using perhaps mkfs.ext4, for example), then to mount the filesystem and access the device through the directory tree.

Other uses include reading the block device to create a disk image, or writing to it to make use of a disk image. Through the device file, such operations can be carried out with ordinary user-space tools.

Pseudo Files

The Linux kernel augments the traditional Unix directory tree with pseudo (or "synthetic") files, in /proc and /sys. These files do not exist on disk, and do not exist on any persistent storage. Instead, their contents are generated dynamically by the kernel as processes interact with them.

For example, /proc/meminfo contains a description of the the system's current memory usage:

$ cat /proc/meminfo
MemTotal: 3887988 kB
MemFree: 1619568 kB
MemAvailable: 2450268 kB
Buffers: 120152 kB
Cached: 984208 kB
SwapCached: 0 kB
Active: 324532 kB
Inactive: 1528196 kB
Active(anon): 9476 kB
Inactive(anon): 1061224 kB
[...]

When cat reads the file /proc/meminfo, the kernel generates the above data, which is then printed to STDOUT.

/proc contains one directory for each process (running program) on the system; therein, each directory contains substantial information about its respective process. /sys contains information useful to only the lowest-level system daemons.[4]

Other File Types:

  • FIFO: Also known as a named pipe, this is a file type that connects one running program to another. "FIFO" stands for First in, first out.
  • Socket: A file that supports interprocess communication across networks.

Michael Kerrisk has called the Unix file interface, "The Universal I/O Model," and for good reason. The four file operations apply to all file types except directories, whose structure (a tree) is protected by the kernel through a distinct set of system calls.

Basic File Operations

and Directories

Basic utilities to manipulate files:

  • touch - Change file timestamps. Can be used to create new and empty files.
  • cp - Copy files and directories
  • rm - Remove files or directories
  • mv - Move (rename) files
  • chown - Change file owner and group
  • chmod - Change access permissions

Utilities to manipulate file contents:

  • cat - Concatenate files and print on the standard output
  • head - Output the first part of a file
  • wc - Print newline, word and byte counts
  • sort - Sort text files

Directories

  • mkdir - Make directories
  • rmdir - Remove empty directories
  • readlink - Print value of symlink or canonical file name

The contents of directories are protected by the kernel. Each directory contains at least two entries: . ("dot"), and .. ("dot dot").

. is a reference to the containing directory itself. It is used by processes as a handle to the current working directory.

.. is a reference to the containing directory's parent directory. Since each (and every) directory has a single parent, the directory structure forms a tree.

Hidden Files

By convention alone, all filenames beginning with . are considered hidden, including . and ... They can be listed with ls by the -a ("all") switch:

$ ls -a

Filesystem Heirarchy

Within any Unix-like operating system, some common themes will be found. For instance, bin directories are found in several places, such as /bin, /sbin, along with /usr/bin and often /home/user/bin as well. These contain binary executable files– programs– such as bash and ls. Other standard directories include /root, the root user's home directory, and /proc, and /dev as mentioned above. Some of these have been standard since Unix's inception, while others have become standard in more recent years.

Aside:

The Filesystem Heirarchy Standard describes the GNU/Linux directory structure, and is maintained by the Linux Foundation. It is intended to promote inter- operability among Unix-like operating systems,[5] and was developed with help from the BSD development community.[6]

Let's begin at the top. Each Unix-like operating system has a single directory, /, which is the root of the directory tree. Every file on the system may be addressed in absolute terms by giving its location relative to this directory.

/home contains users' home directories. As user josh, my home directory is /home/josh. This path is commonly abbreviated to ~. By default, it contains the usual "Music", "Movies", "Pictures" and "Downloads". Per-user configuration files are stored here— as either hidden files or in the directory ~/.config.

The directory /etc contains system-specific configuration files. For example, /etc/passwd contains essential information about each user on the system (but not their password, anymore). In addition, the configuration of the package manager is stored here, and, well, etcetera.

/tmp, /run, and /var each hold transient files of running programs. /boot holds everything required for the boot sequence, including my very own Linux kernel:

$ file /boot/Image
Image:   Linux kernel ARM64 boot executable Image, little-endian, 4k pages

This kernel is loaded in early boot, after which the file is no longer needed.

Software libraries are kind of boring, but essential to modern systems. The basic idea is to factor out repeated code (for instance, C's printf function) not just from the source code, but from executable files as well. This means that, when a bug or security vulnerability is found in printf, any program that uses the function will not have to be recompiled; rather, the appropriate library can be adjusted, and all dependant programs can continue operation in blissful ignorance of the change. Such executable files are said to be dynamically linked, and cannot be expected to run properly without their dependencies. /lib holds libraries.

The last directory we will look at is /usr, which is more interesting. It contains user-shared files, such as man pages (in /usr/share/man) and most of the system's programs (in /usr/bin). These files are shared by all system users, and are mostly read-only.

Several utilities exist to help locate files; one of my favorites is whereis, available in the util-linux package.

Mounting a Filesystem

As mentioned, Unix-like operating systems maintain a single directory tree for the entire system. When a new device is introduced to the machine, its directory tree must be placed in the system's existing directory tree. This operation is known as mounting. $ sudo mount device mountpoint

Superuser priveledges allow us to mount a filesystem anywhere. When a filesystem is mounted to a directory that is not empty, those directory's contents become hidden by those of the new filesystem, and are inaccessible while the filesystem is mounted. Therefore, we typically create an empty directory in the existing directory tree, to serve as the to-be mounted filesystem's root directory.

inodes

In keeping with traditional Unix file semantics, every file on a GNU/Linux system is represented by a unique inode.[7] Inodes are the responsibility of their respective filesystem, wherein each is addressed by a number. Within a filesystem, an inode number identifies an inode, and thereby identifies a particular file.

A directory is a file that associates filenames (strings) with inode numbers:

Remember:

Every path on the directory tree terminates in an inode reference.

$ ls -ia ~/Public/Learning-Linux
645278 ./
384844 ../
775487 assets/
641473 command-line-1-tutorial-introduction.html
645403 _config.yml
775449 _data/
642188 documentation.html
642185 files-and-filesystems.html
641460 finding-files.html
775463 _future/
[...]

Each such association is called a hard link. A hard link is created upon file creation. We can also create new hard links with the ln command, as the following sequence demonstrates:

$ echo -n "Hello " > newfile
$ ln newfile 2file
$ ls -i newfile 2file
2112949 newfile
2112949 2file

The name 2file refers to the same file as newfile. If we modify the file using one name, then we have modified the file referred to by the other:

$ echo "world!" >> 2file
$ cat newfile
Hello world!

A file may have more than one hard link, but when it has zero, it is no longer accessible through the directory tree, and the kernel prepares to delete it.[8]

The Original Unix Filesystem

The inode was implemented literally in the Research Unix filesystem, and formed the basis of its architecture:

In the design, data blocks are of fixed size, and either free or wholly consumed by a file.[9] The filesystem's inodes represent one file each, contain their file's metadata, and contain pointers to data blocks.[9] These nodes are arranged in an array, and can be identified by an index into this array.

The design allows a file to grow or shrink without rearranging other files.[9] In addition, the inclusion of single-, double- and triple- indirection blocks allow the inode to remain of fixed, reasonable size, while supporting large files.[9] Finally, by using direct pointers for the first bytes of a file, and by keeping the inode array in main memory, the design fosters the speedy retrieval of data in small files.[9]

It was the most advanced filesystem of its time, and, as Unix grew in popularity, the semantics of file operations in Unix-like operating systems became dependant upon them. The design was demonstrated in the MINIX filesystem, which in turn served as Linux's pre-1.0 filesystem. The extended filesystem, ext, extended those filesystem's capabilities. With ext's inclusion in kernel 0.96c, Linux began to use its new Virtual File System, a kernel subsystem that provides a uniform filesystem interface.

While each filesystem implements its own functionality, most functionality is common; the Virtual File System maps system calls to filesystem-driver functions. In addition, the VFS imposes inodes on filesystems that do not natively employ them, thereby preserving traditional Unix file semantics across all supported filesystems.

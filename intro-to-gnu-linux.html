Introduction to GNU/Linux

"Those who do not understand UNIX
are condemned to reinvent it, poorly."
Henry Spence, Programmer

An operating system is a collection of software designed to facilitate the use of a computer. To this end, its two main responsibilities are to support running programs— known as processes— and to maintain persistent data— known as files. In addition, operating systems ease the task of creating useful programs by giving processes a predictable environment. Modern, open-source GNU/Linux offers the single best programming environment available today.

The Unix Architecture

GNU/Linux belongs to a large class of operating systems called Unix-like, sometimes referred to as *nix. The Unix operating system had a clear and well-documented architecture, which has been replicated many times in both open-source and proprietary contexts. An overview of its architecture is as follows:

Hardware: At the heart of any machine is its hardware. Each piece of hardware has a software interface defined by the engineers that created it. These interfaces are varied at best.

Aside:

Curiously, hardware is becoming increasingly integrated. Hard-disk controllers have long been integrated into the drive itself, and System-on-a-Chip's (SOC's) are becoming increasingly popular.

Drivers: These interact directly with hardware and provide the kernel with a more-uniform interface to each piece. This software layer represents the deepest and most remote element of any operating system. Although drivers can be installed on an as-needed basis, in GNU/Linux the tendency is to include many drivers, and use only those that are required.

Kernel: While the heart of any machine is its hardware, the heart of any Unix-like operating system is its kernel. The kernel uses drivers to direct the actions of the machine, at the behest of processes. At some 28 million lines of code, this is a large and involved piece of software.

System Calls: In order to permit interaction with the kernel, each Unix-like operating system employs a set of system calls. From a source-code point of view, system calls "look" and "act" like ordinary function calls, but their actions are not restricted to user-space. Rather, they transcend the software hierarchy, often making changes at the hardware level.

Processes: Sometimes called a "user-space process," a process is an executing program.[1] On Unix-like systems, processes run the entire show, and are the only means that a user has to interact with the system. As such, processes are a critical component of a functioning operating system.

The Linux Kernel

Properly speaking, the name Linux refers only to the Linux Kernel. The Linux Kernel was started by Linus Torvalds in 1991, then a graduate student of the University of Helsinki.[2] Following the architecture of the Unix operating system, he began writing an open-source operating system as "...a hobby, won't be big and professional like [the GNU Project]."[3] He soon released it publicly under the GNU General Public License, version 2. Since then, it has become very successful, and has become the most-used Unix-like kernel in existence.

Developing the kernel is not an easy task. The development process must:

  • Facilitate change, so that new features may be incorporated
  • Yield stable kernels, for private and commercial use
  • Accept patches from thousands of unknown and volunteer developers each month

In addition, the kernel development team takes care not to break user-space, so that what did work will continue to work.[4] As such, the development process is somewhat involved. Documentation about it is maintained within the source code.

Linus Torvalds remains the maintainer of the Development branch today. Another major figure in Linux development is Greg Kroah-Hartman (pronounced "Crow-Hartman"). He is currently the maintainer of the Stable branch.

GNU

Development and deployment of Unix-like operating systems is contingent upon some user-space programs, such as a C compiler and an sh-compatible shell. GNU (pronounced gah-new) is well-known for producing high-quality, well-documented free software; they are as "official" as the open-source community gets.

From gnu.org:

“Free software” means software that respects users' freedom and community. Roughly, it means that the users have the freedom to run, copy, distribute, study, change and improve the software.[sic]

The GNU Project was started by Richard Stallman in September 1983,[5] and takes freedom very seriously. In GNU language, free software does not mean merely, "costs nothing." Rather, it is intended to mean free as in freedom.[6] GNU and the open-source community often use the term libre to connote this kind of freedom. It should speak volumes that Linus Torvalds chose to release his budding kernel under a GNU license.

GNU's compiler collection, gcc, contains the de-facto C compiler of the open-source community. This compiler is the only one capable of building Linux kernels.[7] Similarly, their bash is the default shell on these systems. Each tool offered by GNU is well-documented, so look to them for documentation.

A point of note: The GNU Project maintains a list of "free" operating systems on their website. Their criteria is extremely restrictive, disallowing non-free software to exist even in downloadable form on the distributions' package-management repositories. Most Linux distributions are not listed; this is permissiveness on the part of the distribution's maintainer, allowing non-free software to be officially supported, and does not mean that you can't use it.

POSIX

The set of system calls that an operating system provides determines the interface that any user-space process has to the operating system, and thus to the machine. In order to assure software portability across operating systems, it is essential that this interface be (mostly) uniform.

The first Portable Operating System Interface specification was released as an IEEE standard in 1988,[8] and specified the system-call interface available to user-space programs. This standard has been refined since then. Operating systems that abide by this standard are called POSIX Compliant; systems developers general aim for POSIX compliance.[9]

A stronger specification is the Single Unix Specification, which is a superset of POSIX.[10] Certification to this standard is costly, and is generally reserved for proprietary operating systems.

While the system-call interface is not the only means of interacting with the kernel, and properly speaking belongs to the kernel, I feel it a sufficiently important abstraction layer to be depicted separately: Most interactions between processes and the operating system occur through the system-call interface.

Linux Distributions

A distribution is a collection of software that, when installed, yields a functioning operating system. Each will include a kernel, compiled with some drivers, core utilities, and, usually, some other user-space applications (e.g., Firefox). All of this is tested to work and will be packaged in a fashion that facilitates installation (usually, an ISO).

In my opinion, distributions are more similar than dissimilar. Think of them as flavors of the Unix architecture, rather than distinct operating systems. The oldest distribution that's still maintained is Slackware.[11]

One of the key distinctions between distributions is the package management system that is provided. The package-management system is intended to be the user's interface to obtaining, installing and upgrading software; it's similar to a Store on closed-source OS's. While the user is free to install software "by hand," this is generally discouraged because the package-management system allows the centralized and automated maintenance of software.

Distributions differ in philosophy towards software that's appropriate to their operating system. Loosely, newer software has newer features, but is less stable than older software. For an operating system intended to be used on a server (e.g., Debian), the maintainers would tend towards reliable, stable, well-tested and mature user-space software. Conversely, a distribution geared towards development (e.g., Fedora) would employ software which is newer and less-tested.

Some distributions follow a "rolling release" development cycle— notably, Arch. This method of delivering software focuses on obtaining and using the latest version of each. Under the rolling release paradigm, new software releases are expected to work correctly only with the latest software releases. The user is expected to update/upgrade frequently, but is guaranteed that each system-wide update will yield a coherent collection of software.

Linux Mint

I like Linux Mint, but then, I'm a simple guy. This distribution is downstream of Debian, which means that it's a derivative of Debian. It inherits their package-management tool, apt, and the philosophy towards software that this implies. Linux Mint is a characteristically professional operating system, and focuses on productivity. It is extremely new-user friendly, but retains the power of any other Unix-like operating system.

By RealNZZN - Own work, CC BY-SA 4.0, Link

Mint offers the Cinnamon desktop environment; Windows users will feel right at home. I find it to be an attractive desktop, but it's only moderately customizable. Like many users, I just want to get to work, so a desktop environment that stays out of my way helps me do exactly that.

New for Linux Mint 19 is Timeshift. From their github: "Timeshift for Linux is an application that provides functionality similar to the System Restore feature in Windows and the Time Machine tool in Mac OS. Timeshift protects your system by taking incremental snapshots of the file system at regular intervals. These snapshots can be restored at a later date to undo all changes to the system."

Mint isn't perfect: It isn't new and isn't flashy. Changes from release to release are minor, and the software available from Debian's apt is comparatively old. But, it is very stable and very predictable. Linux Mint 20 is a long-term support release, and will be supported until 2025.

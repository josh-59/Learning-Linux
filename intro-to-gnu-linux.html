Introduction to GNU/Linux

"Those who do not understand UNIX
are condemned to reinvent it, poorly."
Henry Spence, Programmer

An operating system is a collection of software designed to facilitate the use of a computer. To this end, its two main responsibilities are to support running programs, known as processes, and to maintain persistent data, in the form of files. In addition, operating systems ease the task of creating useful programs by offering programmers a predictable environment. Modern, open-source GNU/Linux offers the single best programming environment available today.

UNIX

The Unix Operating System was born in late 1969 at Bell Labs, in the shadow of Multics (Kernighan, 2020, pp. 31-34). Multics was an operating system intended to commercialize computing by offering it as a service, similar to telephone or electricity.cd Pubcd Users would connnect their text terminals to a central mainframe, then pay for computation on an as-used basis.

Being a joint venture between MIT, AT&T's Bell Laboratories, and General Electric, Multics was a large and, in many ways, novel, project. It introduced many new operating systems ideas— among them, the hierarchical file structure.[1] For a variety of reasons, the project never realized its dream, and in 1969 Bell Labs abandoned it.

Within Bell Labs, management put operating systems research on the backburner, and refused to purchase new computing equipment.[2][3] After finding a disused and underpowered PDP-7, Ken Thompson rewrote Space Travel for it.[2][3] In the process, he accrued the tools and skills necessary to implement a filesystem he and others at Bell Laboratories had been devising on paper.[3]

At some point in 1969, Ken Thompson realized that he was three weeks from an operating system. All he needed was an exec call, an assembler, a text editor and a shell. In 1970, the new operating system was christened UNIX, as a "somewhat treacherous pun on MULTICS."[3]

Owing in part to the restrictive nature of available hardware, the budding operating system focused on simplicity.[4] In addition, early computer researchers were programmers as well as users, so that their operating system facilitated programming as a means of use. This theme has been echoed throughout Linux's history, and is a key to understanding its success.

Aside

The performance penalty implied by rewriting the kernel in C has been estimated at 20-30% over assembly.

The earliest versions of the new operating system were put to use internally by Bell Labs to assist in patent applications.[2][3] The software then moved away from its nest under obtuse licenses to universities, where its code, though not free, was studied and documented.[2] Finally, Unix became commercially available, to principally to corporate clients.[2]

In 1973, after Dennis Ritchie introduced struct's to the C programming language, Ken Thompson was able to rewrite most of Unix in C (Kerninghan, p. 78); this corresponded to UNIX Version 6. This rewrite is especially important to the history of computing because it spawned the notion of portability: The same software, written in a high-level language, can be compiled to run on different machines. By 1978, Unix was running on several different architectures (Kernighan, p. 140).

The Unix Architecture

Unix spawned a large class of operating systems called Unix-like, sometimes referred to as *nix. Unix had a clear and well-documented architecture; its derivatives share this architecture, in addition to the Unix philosophy. An overview of its architecture is as follows:

Hardware: At the heart of any machine is its hardware. Each piece of hardware has a software interface defined by the engineers that created it. These interfaces are varied at best.

Drivers: These interact directly with hardware and provide the kernel with a more-uniform interface to each piece. This software layer represents the deepest and most remote element of any operating system. Although drivers can be installed on an as-needed basis, in GNU/Linux the tendency is to include many drivers, and use only those that are required.

Kernel: While the heart of any machine is its hardware, the heart of any Unix-like operating system is its kernel. The kernel uses drivers to direct the actions of the machine, at the behest of processes. At some 28 million lines of code, this is a large and involved piece of software.

System Calls: In order to permit interaction with the kernel, operating systems employ a set of system calls. From a source-code point of view, system calls "look" and "act" like ordinary function calls, but their actions are not restricted to user-space. Rather, they transcend the software hierarchy, often propogating changes at the hardware level. Unix-like operating systems have six central system calls: fork, exec, open, close, read, and write.

Processes: Sometimes called "user-space processes," a process is an executing program. On *nix platforms, processes run the entire show. They are the only means that a user has to interact with the system, and the kernel depends on them to direct its actions. Processes are a critical component of a functioning operating system.

GNU

The first serious attempt at producing a free and open-source Unix-like operating system was announced by Richard Stallman in 1983. Stallman was, at the time, a graduate student at MIT's AI Laboratory,[5] and his project was called "GNU," a recursive acronym for GNU's Not Unix.[6]

From gnu.org:

“Free software” means software that respects users' freedom and community. Roughly, it means that the users have the freedom to run, copy, distribute, study, change and improve the software.[sic]

To create an entire operating system is an enormous task, but, "...by 1990 [the GNU Project] had either found or written all the major components except one– the kernel."[6] GNU's own kernel, dubbed Hurd, is, as of 2020, in version 0.9.

Today, the GNU Project is well-regarded for producing high-quality, well-documented free software; they are as "official" as the open-source community gets.

In particular, GNU's compiler collection, gcc, contains the de-facto C compiler of the open-source community. This compiler is the only one capable of building Linux kernels.[7] Similarly, their Bash shell is the default shell on many distributions.

The GNU Project takes freedom very seriously. In their language, free does not mean merely, "costs nothing." Rather, it means free as in freedom.[8] Thus, GNU's software is free in terms of both price and liberty; the open-source community sometimes uses the term libre to connote this kind of freedom.

The Linux Kernel

On August 25, 1991, a Finnish graduate student by the name of Linus Torvalds announced his own free operating system project. Again following the architecture of the Unix operating system, his project was intended to be "...just a hobby, won't be big and professional like gnu[sic]."[9] He soon released it under the GNU General Public License, Version 2.

Aside:

Properly speaking, the name Linux refers only to the Linux kernel.

While the GNU Project ambitiously intended to produce an entire operating system, Linus restricted his attention to creating a kernel. By October, he had Bash, GCC, GNU-Make and other programs running on his creation.[9]

Moreover, while GNU sought professionalism and officiality, Linux was, "...meant for hackers interested in operating systems and 386's."[9] Interest in the Linux kernel flourished, in large part because of the informality of the project: Everybody was invited, if only to tinker with it. It has since become the most successful Unix-like kernel in existence.

The kernel development process has become more structured since 1991, and today uses a "train-schedule" approach. Today, Linus Torvalds remains the maintainer of the Development branch. Another major figure in Linux development is Greg Kroah-Hartman (pronounced "Crow-Hartman"), who is the maintainer of the Stable branch.

Although development of Linux began as a hobby, it is an occupation for some, and serious affair for many. In particular, the kernel development team takes great care not to break user-space,[10] meaning that programs that did work will continue to work. In this way, they ensure the long-term stability of Linux-based platforms.[11]

POSIX

The system calls that an operating system provides determine the interface that any user-space process has to the operating system, and thus to the machine. In order to assure software portability across operating systems, it is essential that this interface be (mostly) uniform.

Aside:

Richard Stallman suggested the name POSIX. It is pronounced pahz-icks.

The first Portable Operating System Interface specification was released as an IEEE standard in 1988,[12] and specified the system-call interface available to user-space programs, among other things. Operating systems that abide by this standard are called POSIX Compliant; systems developers general aim for POSIX compliance.[13]

Posix has been refined since 1988. In addition, a stronger specification has been introduced, and is called the Single Unix Specification.[13] Certification to this standard is costly, and it is generally reserved for proprietary operating systems.

While the system-call interface is not the only means of interacting with the kernel, and properly speaking belongs to the kernel, I feel it a sufficiently important abstraction layer to be depicted separately: Most interactions between processes and the operating system occur through the system-call interface.

Linux Distributions

A Linux distribution is a collection of software that, when installed, yields a functioning operating system. Each includes a kernel, some core utilities, and, usually, some user-space applications (e.g., Firefox). All of this is tested to work and will be packaged in a fashion that facilitates installation, (usually, an ISO).

In my opinion, distributions are more similar than dissimilar. Think of them as flavors of the Unix architecture, rather than distinct operating systems. The oldest distribution that's still maintained is Slackware, first released in 1993.[14]

One of the key distinctions between distributions is the package management system that is provided. The package-management system is intended to be the user's interface to obtaining, installing and upgrading software; it's similar to a Store on closed-source OS's. While the user is free to install software "by hand," this is generally discouraged because the package-management system allows centralized and automated updating.

Distributions differ in philosophy towards the software that's appropriate to their operating system. Loosely, newer software has newer features, but is less stable than older software. For an operating system intended to be used on a server (e.g., Debian), the maintainers would tend towards reliable, stable, and well-tested user-space software. Similarly, a distribution geared towards development (e.g., Fedora) would employ software which is newer and less-tested.

Some distributions follow a "rolling release" development cycle— notably, Arch. This method of delivering software focuses on obtaining and using the latest version of each. Under the rolling release paradigm, new software releases are expected to work correctly only with the latest software releases. The user is expected to update/upgrade frequently, but is guaranteed that each system-wide update will yield a coherent collection of software.

Linux Mint

I like Linux Mint, but then, I'm a simple guy. This distribution is downstream of Debian, which means that it's a derivative of Debian. It inherits their package-management tool, apt, and also Debain's philosophy towards software. Linux Mint is a characteristically professional operating system, and focuses on productivity. It is extremely new-user friendly, but retains the power of any other Unix-like operating system.

By RealNZZN - Own work, CC BY-SA 4.0, Link

Mint offers the Cinnamon desktop environment; Windows users will feel right at home. I find it to be an attractive desktop, but it's only moderately customizable. Like many users, I just want to get to work, so a desktop environment that stays out of my way helps me do exactly that.

New for Linux Mint 19 is Timeshift. From their github: "Timeshift for Linux is an application that provides functionality similar to the System Restore feature in Windows and the Time Machine tool in Mac OS. Timeshift protects your system by taking incremental snapshots of the file system at regular intervals. These snapshots can be restored at a later date to undo all changes to the system."

Mint isn't perfect: It isn't new and isn't flashy. Changes from release to release are minor, and the software available from Debian's apt is comparatively old. But, it is very stable and very predictable. Linux Mint 20 is a long-term support release, and will be supported until 2025.

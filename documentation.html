Documentation

Aside:

In days' past, it was common practice to inspect source code as documentation.

The Free-and-Open-Source Software (FOSS) community centers around projects. At the center of each project is its code. Official documentation, if it exists, is part of the project. Outside the project, there is unofficial documentation. Finally, at the periphery, is opinion. Good, organized documentation is very important to the success of a project. This paper discusses different sources of information, and where they may be appropriate.

Man Pages

By far, the most useful asset in the Linux documentation pool is the manual ("man") pages. These date to the first implementations of Unix,[1] where they were the user's primary source for information about the new operating system. Man pages provide a concise overview of commands and features, with further documentation existing elsewhere. Learn to use man pages early, for they are a way of life.

Man pages are accessed at the command line:

$ man cat

Once within the man utility, you can easily search the content by typing a single forward slash, then your search term:

/<search-term>

Then, enter. Jump to the next result by typing a single n. To quit the man paging utility, type q.

The controls of the man paging utility are simple enough: To move down one line, type j; to move up one line, use k. To move down a screenful, press the spacebar.

Sections

The manual is divided into seven different sections, as shown below. Here we will use the standard convention for specifying specific man pages: The name of the page is followed by its section number, which is placed in parenthesis. For example, cat(1) refers to the manual page for the command-line utility, while printf(3) refers to the manual page of the C library function.

Manual Sections:

  1. Shell Commands
  2. Systems Calls
  3. C Library Functions
  4. Special Files
  5. File Formats and Conventions
  6. Games
  7. Miscellaneous

The man utility searches the sections in order, and so defaults to the page with the lowest section number. Hence,

$ man printf

displays printf(1), if it exists. To restrict man's search to section three, we can use either

$ man 3 printf

or

$ man printf.3

Official Documentation

Official documentation originates from the group or organization that is producing the work; it's part of the project itself. It is suitably positioned to be both exhaustive and authoritative over the subject. Think of official documentation like the primary sources from history class.

Like the projects they are part of, official documentation is usually community-driven. Moreover, it is typically found in web-page format.

Several examples spring to mind:

  • The Linux Kernel Documentation— A book-like website documenting the latest kernel.
  • Arch Linux's Wiki— The official documentation of the Arch distribution. This is an excellent resource, and its material often applies to distributions outside of Arch.
  • GNU's Official Docs— Excellent documentation. It is both supremely authoritative and exhaustive, but owing to its precision, and the precision of the work they do, it can be difficult to read.
  • Godot Open Source Game Engine— Godot is an MIT-licensed open source game engine. Its official documentation is community-driven.
  • Rust's Library Documentation— This documentation scheme is fantastic. It was created with the developer in mind, and is easy to navigate, easy to read, and makes the libraries easy to comprehend. Moreover, facilities are included in the compiler itself, which permit documentation extraction from source-code comments. It's easy, fast and clean.

Unofficial Documentation

Most documentation is from unofficial, secondary sources. Examples include this website, blogs, question/answer sites, Youtube videos, podcasts, books, etc.. Think of unofficial documentation as commentary on the subject at hand. It can be useful when the subject is difficult, but official documentation should still be your reference.

Examples of good, unofficial documentation:

  • The Linux Documentation Project— From the website: "TLDP is short for The Linux Documentation Project, an organization of volunteers authoring, reviewing and managing documents about the Linux operating system. Documents basically come in two formats based upon their length. The shorter ones generally are called HOWTOs (or mini-HOWTOs, if they are really short), the longer documents, called guides, deal in-depth with a Linux feature."[2]

    TLDP is not maintained anymore, so that its content is sometimes out-of-date (but still very good).
  • Debian Administrator's Handbook— A free book written by RaphaÃ«l Hertzog and Roland Mas, two Debian developers. It was originally written in French, and has been translated into four other languages, with twenty(!) more in-progress.
  • Linux From Scratch — From their website, "Linux From Scratch (LFS) is a project that provides you with step-by-step instructions for building your own customized Linux system entirely from source."[3] It's a thing.
  • Linux Kernel in a Nutshell— A free book written by Greg Kroah-Hartman. He says, "I want this book to help bring more people into the Linux kernel development fold. The act of building a customized kernel for your machine is one of the basic tasks needed to become a Linux kernel developer. The more people that try this out, and realize that there is not any real magic behind the whole Linux kernel process, the more people will be willing to jump in and help out in making the kernel the best that it can be."[4]

Nonfree Books

In the main, I recommend staying away from non-free *nix books. There are several reasons for this.

First, non-free books are almost always secondary sources, and hence, not authoritative. The books you want to read are by those people closest to the FOSS community, and they usually (but not always) write free books. If it is free, they often offer a paperback copy for sale, and/or a means to donate/support.

Second, there's just too many of them, and very few are worthwhile. For instance, I bought a handbook on UNIX system administration that looked very promising, but taught me precious little. In fact, the free Debian Administrator's Handbook proved a more succinct, more authoritative resource.

Third, they tend to be out-of-date. For example, I really liked the premise of Understanding UNIX/LINUX Programming: A Guide to Theory and Practice . It walks you through programming in Linux by recreating common command-line tools, such as cat and ls, but it was so out-of-date by the time I read it that the information in it couldn't be trusted. It'd be great to see this rewritten in a modern context.

Some exceptions:

  • The Linux Programming Interface— This is the definitive text on the programming interface of the Linux Kernel. If you want all the gory details of the system-call interface offered by the Linux kernel, as well as sound explanations of its internals, this is your book. Its writer, Michael Kerrisk, is currently the maintainer of the Linux Man Page project (see above), and his work is simply terrific.
  • Computer Systems: A Programmer's Perspective — From its preface: "This book (known as CS:APP) is for computer scientists, computer engineers, and others who want to be able to write better programs by learning what is going on "under the hood" of a computer system."[5]

How-to

It's worthwhile to take a step back and consider the process of learning Linux itself. In my experience, the best method of learning Linux is to focus on the underlying concepts: When the concept is understood, one can usually "feel" one's way through rectifying the situation at the command line. Let's take an example.

Suppose you are working at the command-line, and you have a question about copying files. Knowing that the command cp copies a file, you type man cp, and the manual page appears. When you type apropos copy, looking for other commands that can copy a file, it shows no results. You know this to be an error, because apropos is supposed to search manual pages and list all that match "copy," which would certainly include cp.

  1. Figure out what your question is.

    Initially, your question might be Why doesn't this work?— and that's totally fine. Figuring out your question can be more difficult than you'd guess, because you might not know how to phrase it. Try to be specific: When dealing with computers, things are very precisely defined.

    Here, the question is, Why isn't apropos returning any results?

  2. What would answer your question?

    This strikes you as such an obvious error that any reasonable documentation should answer your question— and you would be right. So, you look first into the man pages, with man apropos. There, you discover that apropos doesn't search the man pages directly, but instead searches a database of information about them. (It does this because it's much faster.)

    Reasonably, you suspect that there might be an error with this database, such as, It doesn't exist. A little further down the page, you read, "The database searched by apropos is updated by mandb. You try mandb, and voila! apropos works as it should.

  3. Be Patient.

    Databases like this pop up in other contexts, as well. For instance, in package management, a common practice is to maintain a local database about available packages, so that the user can search for packages without bothering public servers. As such, updating these databases (on Debian/Ubuntu/Mint, with apt update) is a precursive step to obtaining the latest packages.

    The point here is that you should take the time to try to learn underlying concepts, because it usually pays off in the long run. Skill accrues. Answer one question at a time.

Documentation

"Understanding is the key to success with Linux."
Linux System Administrator's Guide

Linux is a very large subject. Although there are many sources of documentation, they are not, in general, well-organized, so that finding the right information can be hard. This makes good information harder-won than it ought to be, and implies that finding good (i.e., appropriate) sources is a skill in itself.

This paper gives an overview of available resources, some examples of each, and useful pointers, where appropriate. First, though, we make explicit the process of determining appropriate sources.

How-to

Suppose you are working at the command-line, and you have a question about copying files. Knowing that the command cp copies a file, you type man cp, and the manual page appears. When you type apropos copy, looking for other commands that can copy a file, it shows no results— you know this to be an error, because apropos is supposed to search manual pages and list all that match "copy".

  1. Figure out what your question is.

    Initially, your question might be Why this no work, and that's totally fine. Figuring out your question can be more difficult than you might guess, because you might not know how to phrase it. Try to be specific: When dealing with computers, things are very precisely defined.

  2. What would answer your question?

    This strikes you as such an obvious error that any reasonable documentation should answer your question (and you would be right). So, you first look into the man pages, with man apropos.

    There, you discover that apropos doesn't search the man pages directly, but instead searches a database of information about them (it does this because it's much faster). Reasonably, you suspect that there might be an error with this database (such as, it doesn't exist).

    A little further down the page, you read, "The database searched by apropos is updated by mandb. You run the program, and voila! apropos works as it should.

  3. Revise.

    Do you need more information? Or less? Should it be a simpler source, or more involved? You might find that your practical knowledge is lacking some kind of body; in this case, you are looking to frame your knowledge in a more-general theory.

  4. Be Patient.

    Databases like this pop up in other contexts, as well. For instance, in package management, a common practice is to maintain a local database of package metadata, so that the user can search for packages without bothering public servers. As such, updating these databases (on Debian/Ubuntu/Mint, with apt update) is a precursive step to obtaining the latest packages.

    The point here is that you should try to take the time to learn underlying concepts, because it usually pays off in the long run. Skill accrues. Answer one question at a time.

By far, the worst habit to get into is expecting to be spoon-fed solutions as though by technical support. Imagine, for a second, that you had *instantly* appealed to Google, and there you found a response such as, "Just run mandb." This is correct, and it solves your problem, but you've learned nothing, and if you continue "learning" this way, you'll be stuck memorizing trivial idioms.

Linux is written by programmers, for programmers. It is very pure with respect to computer science, and being a competent user necessarily requires some CS know-how. If you're not interested in learning that, then find somewhere else to go (but if you are interested, stick around!).

Man Pages

By far the most useful asset in the Linux documentation pool is the manual ("man") pages. Man pages date back to the first implementations of Unix,[1] where they were the user's primary documentation of the newly-minted operating system. They often included examples, hints and useful suggestions, such as, "cat file is about the easiest way to print a file."

Today, the man pages are maintained by Michael Kerrisk, and often include examples, hints, and useful suggestions, such as, "These are not the interfaces you are interested in." Learn to use man pages early, for they are a way of life.

Manual Sections:

  1. Shell Commands
  2. Systems Calls
  3. C Library Functions
  4. Special Files
  5. File Formats and Conventions
  6. Games
  7. Miscellaneous

Man pages are accessed at the command line so:

man app

You can search through available man pages with the apropos command, as in,

apropos decompress

or

apropos extract

Technically speaking, this utility searches the single-line, "short description" of each man page, and prints out the names and descriptions of programs that match your search (man apropos for more information).

Once within the man utility, you can easily search the content by typing a forward slash, then your search term:

/<search-term>

Then, enter. To jump to the next result, type a single n.

Many of the controls of the man-pages coincide with those of the less paging utility, and the vim text editor, so they're worth learning. For instance, to move down one line, type j. To move up one line, type k. And, move down a screenful by hitting the spacebar. Why the crazy controls? It keeps your fingers on the home row.

The manual is structured into sections, as shown above. In particular, Section 1 contains entries for user commands, such as cat, man, mv and cp, while Section 3 contains the manual pages for C library functions, such as getline(), printf() and strtol(). When using apropos, one can restrict the search to a particular section with the -s switch:

apropos -s 3 printf

Otherwise, apropos lists them all.

In addition, each manual page is organized into a common format. For instance, each page has a "See Also" section near the bottom, which lists related commands/library functions.

Official Documentation

Official documentation originates from the group or organization that is producing the work; it is part of the project itself. It is suitably positioned to be both exhaustive and authoritative over the subject. Think of offical documentation like the primary sources from history class.

In some sense this is the only form of true "documentation," properly speaking; everything else is commentary. Like the projects they are part of, official documentation is usually community-driven. Moreover, it is typically found in web-page format.

Several examples spring to mind:

  • The Linux Kernel Documentation— A book-like documenting the latest kernel.
  • Arch Linux's Wiki— The official documentation of the Arch distribution. This is an excellent resource, and its material often applies to distributions outside of Arch.
  • GNU's Official Docs— Excellent documentation. It is both supremely authoritative and exhaustive, but owing to its precision, and the precision of the work they do, it can be difficult to read. Also, note that they offer (the same) documentation in multiple formats: HTML, PDF, and through their own info(1) utility.
  • Godot Open Source Game Engine— Godot is an MIT-licensed open source game engine. Its official documentation is community-driven.
  • The Rust Progamming Language— A book written by Steve Klabnik and Carol Nichols with updates from the community, this is part of the official documentation of the Rust language. I think it's very well-written, especially notable for its technical excellence.

Unofficial Documentation

Most documentation is from unofficial, secondary sources. Examples include this website, blogs, question/answer sites, Youtube videos, podcasts, books, etc..

Think of unofficial documentation as commentary on the subject at hand. It can be useful when the subject is difficult (for instance, Bash), but official documentation should still be considered your reference.

Examples of good, unofficial documentation:

  • The Linux Documentation Project— This is very close to being official documentation, in virtue of the nature of the community that created Linux. However, it is not officially coupled to any project, and so is best considered a secondary source. Moreover, its scope is larger than the kernel. TLDP is not maintained anymore, so that its content is often out of date (but still very good).
  • Debian Administrator's Handbook— A free book written by RaphaÃ«l Hertzog and Roland Mas, two Debian developers, it was originally written in French, and has been translated into four other languages, with twenty(!) more in-progress.
  • Linux From Scratch — From their website, "Linux From Scratch (LFS) is a project that provides you with step-by-step instructions for building your own customized Linux system entirely from source."[2] It's a thing.

Note: GNU/Linux is an open-source community; respect it by avoiding commercial (.com) sites that "offer" advice and documentation about Linux. Also, StackOverflow is not the appropriate avenue for bug reports.

Nonfree Books

In the main, I recommend staying away from non-free *nix books. There are several reasons for this.

First, there's just too many of them, and very few are worthwhile. For instance, I bought a handbook on UNIX system administration that looked very promising, but taught me precious little. In fact, the free Debian Administrator's Handbook proved a more succinct, more direct and more appropriate resource.

Aside:

My aim here is to warn the reader, not to make light of other peoples' work.

Second, non-free books are almost always secondary sources, and hence not authoritative; moreover, there's a good chance that the one you try will not be the best resource.

Third, they tend to be *somewhat* out-of-date. For example, I really liked the premise of Understanding UNIX/LINUX Programming: A Guide to Theory and Practice. It walks you through programming in Linux by recreating common command-line tools, such as cat(1) and ls(1), but it was so out-of-date by the time I read it that the information in it couldn't be trusted. It'd be great to see this rewritten in a modern context.

Some exceptions:

  • The Linux Programming Interface— This is the definitive text on the programming interface of the Linux Kernel. If you want all the gory details of the system-call interface offered by the Linux kernel, as well as sound explanations of its internals, this is your book. Its writer, Michael Kerrisk, is currently the maintainer of the Linux Man Page project, and his work is simply terrific.
  • The C Programming Language, Second Edition— Written by the creators of the language itself, this is the definitive resource on C.

In general, the people who ought to be writing books are those that work in the FOSS community, and those people usually write free books. They often offer a paid-for paperback copy, or other means to donate.

Tutorials

Tutorials are a great form of documentation, because they are informative, concise, and relevant. A tutorial jumps right in to the subject at hand.

Good examples of good tutorials:

  • Make an Action RPG in Godot 3.2— A Youtube tutorial for building your own Godot 3.2 game.
  • Learn Markdown— A tutorial guide to markdown.
  • Making Your Own Ruby Gems — Walks you through the creation of a Ruby software package (a.k.a., 'gem').

News

What community would be complete without a means of staying up-to-date? Most projects maintain an official blog.

  • Level One Techs— Videos, mainly, but love their content.
  • USENIX ;login:— Non-free, though well-written. Publications more than a year old are available freely.
  • Linux Weekly News— Non-free.
